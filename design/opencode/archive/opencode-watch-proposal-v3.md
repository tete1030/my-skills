# OpenCode 会话监控与执行方案提案（V3）

## 为什么要出 V3

V2 的核心思路是：

- 用本地独立 watcher 做核心监控
- 用 cron 做低频兜底和通知
- 只在异常或完成时让 agent/subagent 出手

这个方向在**成本控制**和**机械可靠性**上是合理的，
但它有一个你刚刚指出的关键问题：

> **执行过程太像后台黑盒，不利于用户看到持续进展，也不利于和主 session 中的讨论自然融合。**

所以 V3 的重构目标不是简单调参数，而是要把体系从：

- **后台 watcher 为中心**

改成：

- **主 session 为中心**

也就是说，后续设计应优先保证：

1. 用户能在主 session 中看到持续进展
2. agent 的操作与讨论在同一条叙事链中发生
3. 定时巡查 / 事件巡查 / 人工讨论共享同一主上下文
4. 机械化组件仍然存在，但退居为“输入层/辅助层”，而不是“主执行层”

---

## V3 的核心结论

### 总原则
**让执行过程发生在主 session 中。**

不是：
- 后台 watcher 自己跑完整闭环，主 session 只收结果

而是：
- 后台组件只负责提供触发和增量信息
- **真正的决策、动作、进展汇报，尽量回到主 session 内完成**

这样用户看到的不是“黑盒后台代理”，而是：
- agent 正在做什么
- 为什么这么做
- 现在进展到哪一步
- 是否需要用户介入
- 是否发现偏差并做了修正

---

# 一、V3 的设计原则

## 1. 主 session 是控制面，也是叙事面
主 session 不只是聊天窗口，而应当成为：

- **控制面**：决策在哪发生
- **执行面**：关键动作在哪发生
- **叙事面**：进展如何被持续呈现给用户

也就是说，主 session 既承担“思考与决策”，也承担“对用户可见的过程输出”。

---

## 2. 机械组件降级为输入层，不再做主导编排
事件监听器、轮询器、状态缓存仍然有价值，
但它们的职责要收缩为：

- 提供触发
- 提供增量
- 提供去重与连续性
- 降低 token 消耗

它们不应成为“真正的执行主体”。

---

## 3. 可见性优先于纯粹的后台优雅
V2 更偏向系统工程上的干净边界；
V3 则明确偏向**用户体验和可见过程**。

这意味着即使某些动作从纯系统设计看“放后台更优雅”，
如果它会让用户感到：
- 看不到进展
- 不知道 agent 在干嘛
- 讨论与执行割裂

那就不是优先方案。

---

## 4. 定时巡查仍然要触发 agent，但应尽量服务于主 session 连续性
V2 里已经确定：

- no-change 也应触发 agent 判定
- 触发、动作、上下文沉淀要分离

V3 保留这个判断，
但把“判定层”放回主 session 视角来理解：

> **每次定时巡查 / 事件巡查，原则上都应给主 session 一个决策机会。**

是否最终形成用户可见文本、是否沉淀详细上下文，仍然可以再分层控制。

---

# 二、V3 的系统模型

## 1. Main Session Agent：核心主体
主 session 中的 agent 负责：

- 理解用户当前目标
- 结合最近讨论和历史上下文做判断
- 接收定时巡查 / 事件巡查输入
- 决定是否动作
- 输出进展汇报
- 发起追问、纠正、中断、补救等动作

这是整个系统真正的“脑”。

---

## 2. Trigger Layer：触发输入层
这一层可以来自：

- 定时巡查
- 事件监听
- OpenCode API 状态变化
- 人工用户消息

它的职责不是替代 agent 决策，
而是把信息整理成**紧凑、结构化、可增量消费的输入**，送给主 session。

它更像“传感器”，不是“后台总控”。

---

## 3. State Store：连续性支撑层
状态存储依然需要，但用途改变为：

- 保存增量锚点
- 去重
- 记录最近一次判定结果
- 记录最近一次已通知状态
- 控制 no-change 噪音

它不应该承载完整业务流程，
也不应该把真正的执行逻辑藏在后台。

---

## 4. Optional Subagent：重活外包层
subagent 仍然可以存在，
但用途更明确：

- 承担长耗时、重执行的具体工作
- 或承担复杂诊断分析

但其结果应尽快回流主 session，
由主 session 负责继续叙事、解释和编排。

换句话说：

- **可以外包执行片段**
- **不能把整个体验外包成黑盒**

---

# 三、V3 的推荐架构

## 方案总述
**主 session 中心型编排 + 轻量触发层 + 外部状态连续性 + 必要时子执行单元**

具体分成四层：

### A. 主 session
负责：
- 所有关键判断
- 面向用户的进展汇报
- 与用户讨论自然融合

### B. 触发层
负责：
- 定时触发
- 事件触发
- 整理紧凑状态输入

### C. 状态层
负责：
- 增量锚点
- 去重
- no-change 抑制
- 恢复能力

### D. 子执行层
负责：
- 重活
- 长任务
- 复杂分析

---

# 四、与 V2 最大不同之处

## V2 的核心观念
- watcher 是核心
- main session 更像通知面

## V3 的核心观念
- **main session 才是核心**
- watcher / poller / state store 都只是辅助

这意味着：

### 以前
`watcher -> 判断 -> 通知主 session`

### 现在
`trigger -> 向主 session 提供增量输入 -> 主 session 判断并执行`

差别很大。

V2 中主 session 更像结果接收方；
V3 中主 session 是真正的行动现场。

---

# 五、定时巡查与事件巡查在 V3 中怎么工作

## 1. 定时巡查
定时巡查仍然要保留，
但它的作用变成：

- 定时给主 session 注入一次“决策机会”
- 让主 session 看到当前压缩状态和增量
- 由主 session 判断是否需要继续动作或汇报

### 关键点
定时巡查不是为了后台自己闭环，
而是为了让主 session 保持持续参与。

---

## 2. 事件巡查
事件巡查仍然是最快的信号源，适合：

- 结束
- 失败
- 卡住
- 权限请求
- question/request for input
- todo/checkpoint 显著变化

但事件本身不应直接替代主 session 判断。

它应做的是：
- 捕捉变化
- 生成 delta
- 把 delta 喂给主 session

---

## 3. no-change 策略
V2 的判断仍成立：

- no-change 也应该触发判定

但 V3 需要额外考虑“用户可见性”。

因此建议拆成两层：

### 决策层
- no-change 仍进入主 session 判定

### 呈现层
- 不一定每次 no-change 都发一大段可见消息
- 但应保留一种**可见、低噪音的进展存在感**

例如：
- 长时间无变化时，定期给出简短状态汇报
- 连续多个 no-change 可压缩为一次“仍在监控，无显著变化”
- 避免完全黑盒失联
- 也避免每轮都刷屏

### 推荐策略
把 no-change 分成三类：

1. **短间隔 no-change**
   - 进入判定
   - 通常不单独发消息

2. **累计一段时间的 no-change**
   - 发一条简短进展汇报
   - 例如“已连续检查 N 次，暂无显著变化，继续监控中”

3. **no-change 但触发了策略动作**
   - 即使没变化，agent 仍决定追问/纠偏/停止/升级
   - 这时应明确在主 session 中可见化

---

# 六、为什么主 session 中心型更适合这个场景

## 1. 用户能看到 agent 正在做事
这比“最后丢一个结果”更重要。

因为很多时候用户关心的不只是结果，
而是：
- agent 是否还活着
- agent 是否在推进
- agent 是否理解了刚才的新讨论
- agent 是否正在根据新情况调整策略

这些都需要过程可见。

---

## 2. 讨论与执行可以自然交织
如果执行在后台黑盒中，
就会出现两个世界：

- 前台讨论世界
- 后台执行世界

它们之间只能靠“通知”连接。

而放在主 session 中，
则是同一个世界：

- 用户提意见
- agent 立刻吸收
- 后续巡查与决策自然延续这个讨论

这更符合“有机融合”的目标。

---

## 3. 主上下文更容易保留真正重要的信息
如果关键判断都发生在后台，
主 session 只收到摘要，
很多微妙的上下文关系会丢掉。

而 V3 让关键判断尽量回到主 session，
能更好保留：
- 当前任务为什么这样定义
- 用户刚补充了什么
- agent 为什么改变了策略
- 某次 no-change 为什么反而触发了动作

这些信息对后续连续性很重要。

---

# 七、V3 下主 session 应该怎么“看起来像在执行”

这里不是要求每次都输出很多字，
而是要求用户能感受到：

> **agent 持续在主 session 里推进，而不是偷偷在别处做完。**

因此建议主 session 的输出分为三类：

## 1. 启动型输出
当开始监控/执行时：
- 明确当前目标
- 明确下一步要做什么
- 明确会持续回报进展

## 2. 进展型输出
当状态变化或达到汇报节奏时：
- 用短消息回报阶段推进
- 不需要长篇推理
- 重点是“我现在到哪了”

## 3. 动作型输出
当发生重要决策时：
- 明确说明要做什么
- 为什么做
- 是否需要用户介入

也就是说，主 session 中应保留一条**连续、可见、低噪音的行动轨迹**。

---

# 八、V3 下状态存储的角色

状态存储依然需要，
但其角色不再是“后台替代主 session 的流程容器”，
而是“主 session 的增量辅助器”。

建议记录：

- `lastSeenMessageId`
- `lastCompletedMessageId`
- `lastTodoDigest`
- `lastUpdatedMs`
- `phase`
- `status`
- `lastDecision`
- `lastVisibleUpdateAt`
- `consecutiveNoChangeCount`
- `lastNotifiedState`

其中新增最关键的是：

- `lastVisibleUpdateAt`
- `consecutiveNoChangeCount`

因为 V3 不是单纯追求“少说话”，
而是追求：

- 不黑盒
- 不刷屏
- 有节奏地可见

---

# 九、V3 对 no-change 的最终建议

这是本轮讨论里最关键的重构点之一。

## 保留的部分
- no-change 仍然要触发 agent 判定
- 触发、动作、记忆沉淀仍应分离

## 调整的部分
V2 更偏向：
- no-op 尽量不进入主会话

V3 改成：
- **判定应在主 session 视角发生**
- **但 no-change 的可见输出采用节奏控制，而不是每轮都发**

### 推荐规则
1. 每轮定时巡查都给主 session 一个决策机会
2. 若无动作且距离上次可见更新很短，可静默
3. 若连续 no-change 达到阈值，应给用户一条短进展
4. 若 no-change 本身触发了动作，则必须可见化

这样可以同时满足：
- 连续性
- 可见性
- 上下文控制

---

# 十、V3 对 subagent / 后台执行的态度

V3 不是完全否定后台执行，
而是明确它的边界。

## 适合放后台的
- 很长的具体执行步骤
- 重计算 / 重 IO / 重外部交互
- 复杂分析

## 不适合放后台的
- 整体流程编排
- 关键策略判断
- 主要进展叙事
- 与用户讨论强相关的持续决策

换句话说：

> **后台可以做“工”，主 session 应该做“指挥、解释、汇报、修正”。**

---

# 十一、V3 的 MVP 推荐实现

如果现在就要落地，最推荐的 MVP 不是彻底重写系统，
而是把现有 V2 架构重排为：

## 1. 保留触发层和状态层
保留：
- event/poll 获取增量
- state store 做去重和连续性

## 2. 改变决策流向
从：
- trigger -> watcher 决策 -> 通知用户

改成：
- trigger -> 生成压缩输入 -> **主 session agent 决策** -> 输出进展 / 动作

## 3. 把主 session 设为默认进展出口
默认情况下：
- 阶段推进
- 异常
- 提问
- 长时间 no-change 后的状态汇报

都应优先回到主 session 呈现。

## 4. 仅把重活外包
如果某个动作真的很长：
- 可以临时外包给 subagent
- 但主 session 负责宣布开始、同步关键进展、解释结果

---

# 十二、V3 的完整流程

## 1. 用户提出目标
主 session 明确目标、预期阶段和汇报方式。

## 2. 系统启动触发层
事件监听 / 定时巡查开始工作，
但它们只负责收集增量，不做闭环主导。

## 3. 每次触发都整理成紧凑输入
例如：
- 最新 phase
- 最近 checkpoint
- 是否有 permission/question
- 是否连续 no-change

## 4. 主 session agent 接收这些输入并判定
决定：
- 继续观察
- 输出进展
- 发起纠正
- 提问用户
- 停止
- 升级到 subagent

## 5. 输出进入主 session
用户持续看见：
- agent 现在在做什么
- 是否有变化
- 是否需要自己介入

## 6. 必要时外包重活
subagent 做具体重任务，
但结果尽快回流主 session。

---

# 十三、V3 的一句话结论

最合理的下一版不应该继续把 watcher 放在系统正中央，
而应该改成：

> **以主 session 作为执行与决策中心；让定时巡查和事件巡查只提供增量输入；让用户在主 session 中持续看到 agent 的进展、动作与修正，而不是只看到后台黑盒结果。**
