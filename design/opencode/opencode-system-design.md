# OpenCode 主 Session 中心执行方案

## 目标

设计一套适合 OpenClaw 编排的 OpenCode 执行与巡查机制，同时满足以下要求：

- 用户能在主 session 中持续看到 agent 的进展与动作
- 巡查结果能和主 session 中的讨论自然融合
- 定时巡查与事件巡查共享连续上下文
- no-change 巡查也能触发有效判断
- 系统避免过度黑盒，也避免无意义刷屏
- 机械组件负责降低成本，但不替代主 session 的核心决策地位

---

## 总体原则

### 1. 主 session 是执行中心
主 session 不只是聊天入口，而应同时承担：

- **决策中心**：关键判断在这里发生
- **执行可见面**：关键动作和进展在这里体现
- **讨论融合面**：用户补充的新想法能够立刻进入后续判断

因此，系统不应把完整执行过程藏在后台，仅把结果回传给主 session。

### 2. 触发层是输入层，不是主导层
定时巡查、事件监听、API 轮询等组件仍然有必要，但它们的角色应限制为：

- 捕捉变化
- 提供增量
- 做去重与连续性支撑
- 控制 token 成本

它们不应成为真正的执行主体，也不应替代主 session 的持续判断。

### 3. 可见性优先于纯后台优雅
对于这个场景，用户体验比单纯的后台闭环更重要。系统需要让用户感受到：

- agent 正在持续推进
- agent 正在根据新情况调整
- 当前状态与前面的讨论是一体的

所以方案应优先保证过程可见，而不是把大部分动作做成后台黑盒。

---

## 系统结构

### 1. Main Session Agent
主 session 中的 agent 是整个系统的核心主体，负责：

- 理解用户目标
- 接收巡查输入
- 基于当前上下文做判断
- 决定是否继续观察、汇报、纠偏、提问、停止或升级处理
- 将关键进展以可见方式反馈给用户

### 2. Trigger Layer
触发层负责提供输入，来源包括：

- 定时巡查
- 事件巡查
- OpenCode API 状态变化
- 用户在主 session 中的新消息

它的输出应是结构化、紧凑、可增量消费的信息，而不是替主 session 完成决策。

### 3. State Store
状态存储负责承载连续性，主要用途是：

- 保存增量锚点
- 避免重复读取与重复通知
- 记录最近一次判定结果
- 跟踪连续 no-change 次数
- 控制可见进展的节奏

它用于支撑主 session 的连续判断，而不是在后台独立承载完整业务流程。

### 4. Optional Subagent
subagent 可以存在，但边界必须明确：

- 适合承担重执行、长耗时、重 IO 或复杂分析任务
- 不适合承担完整流程编排
- 不适合作为主要进展叙事者

后台单元可以做“工”，但主 session 应负责“指挥、解释、汇报、修正”。

## 用户输入作为控制面

用户的额外输入要求——例如：

- 调整目标
- 补充约束
- 改变汇报方式
- 要求暂停 / 继续 / 停止
- 指定执行应更多发生在主 session 中

不需要通过额外 UI 完成。
系统应直接把**聊天中的自然语言输入**视为一种统一的控制输入。

这是一种**内部抽象层**，不是新的面板、表单或配置页面。
用户继续通过聊天表达要求，系统内部再把这些要求解释成结构化控制状态。

### 控制输入的地位
用户输入虽然在入口形式上和定时巡查、事件巡查一样，都会触发一次新的判定，
但其本质不是普通状态输入，而是**高优先级控制输入**。

可以把它理解为：

- 定时巡查 / 事件巡查回答：**发生了什么**
- 用户控制输入回答：**现在应该怎么做**

因此，用户输入在优先级上应高于后台默认策略，也高于普通巡查结果。

### 数据流向
用户额外输入进入系统后，建议按以下流向处理：

1. **进入主 session**
2. **由主 session agent 解释为结构化控制项**
3. **写入状态层，成为后续巡查和执行都要遵守的控制状态**
4. **向执行侧传播，影响汇报、巡查、后台执行和后续判断**

也就是说，用户输入不是只在当前这条消息生效，而应成为后续系统行为的控制依据。

### 控制项建模
这里不建议使用多个语义重叠的 boolean，例如：

- `preferMainSessionVisibleExecution`
- `avoidBlackBoxBackgroundExecution`
- `integrateDiscussionAndExecution`

这类字段高度重叠，不够正交。

更好的表示方式有两种。

#### 方式 A：一个总模式

```json
{
  "executionMode": "main_session_centered"
}
```

它隐含的语义是：

- 关键决策优先发生在主 session
- 过程默认应对用户可见
- 讨论与执行保持紧密融合
- 后台执行只能做辅助，不应成为黑盒主流程

#### 方式 B：拆成更正交的维度

```json
{
  "executionCenter": "main_session",
  "progressVisibility": "visible",
  "discussionCoupling": "tight"
}
```

这三个维度分别回答：

- **executionCenter**：主要在哪儿执行/决策
- **progressVisibility**：过程是否默认对用户可见
- **discussionCoupling**：用户在聊天中的新增讨论是否应直接进入执行上下文

相比多个重叠布尔量，这样的建模更清晰，也更便于后续扩展。

### 建议采用的简化方案
在当前阶段，建议优先使用单一总模式：

```json
{
  "executionMode": "main_session_centered"
}
```

如果以后需要更精细的控制，再拆成三条正交维度。

## 定时巡查与事件巡查

### 1. 定时巡查的作用
定时巡查的目的不只是发现变化，而是给主 session 一个周期性的决策机会。

每到定时点，系统都应让 agent 获得一次判断机会，即使本轮没有显式变化，也不能在触发层直接跳过。

### 2. 事件巡查的作用
事件巡查适合捕捉：

- 会话结束
- 失败
- 卡住
- 权限请求
- question/request for input
- todo 或 checkpoint 的显著变化

事件层负责尽快捕捉这些变化，并将其整理为增量输入交给主 session，而不是直接替主 session 完成闭环。

### 3. 两类巡查必须共享连续性
定时巡查与事件巡查应共享同一份状态基础，例如：

- `lastSeenMessageId`
- `lastCompletedMessageId`
- `lastTodoDigest`
- `lastUpdatedMs`
- `phase`
- `status`
- `lastDecision`
- `lastVisibleUpdateAt`
- `consecutiveNoChangeCount`
- `lastNotifiedState`

这样系统的连续性来自统一的外部状态，而不是寄希望于每次巡查都重新理解完整上下文。

## no-change 策略

### 1. no-change 也应触发判定
no-change 只表示本轮没有检测到显式变化，不代表系统不需要做判断。

即使没有新变化，agent 仍可能基于整体上下文决定：

- 继续观察
- 发起追问
- 做纠偏或补救
- 中断或停止某种流程
- 升级到更强处理方式

因此，no-change 巡查也必须进入判定层。

### 2. 触发、动作、沉淀应分离
需要明确区分三件事：

1. 巡查是否发生
2. agent 是否形成动作
3. 是否将本轮内容沉淀为上下文或记忆

建议原则：

- **巡查触发**：始终发生
- **agent 判定**：始终发生
- **上下文沉淀**：按价值决定

### 3. no-change 的可见性应节奏化
V3 不是要求每次 no-change 都发一条消息，而是要求用户不要感到系统完全失联。

建议策略：

- **短间隔 no-change**：进入判定，但通常不单独发消息
- **累计一段时间的 no-change**：压缩成一次简短进展汇报
- **no-change 但触发了动作**：必须在主 session 中可见化

这样既避免刷屏，也避免黑盒失联。

## 主 session 中的可见进展

用户需要看到的是持续推进的过程，而不是最后一次性收到结果。因此主 session 的输出应有明确分层。

### 1. 启动型输出
开始执行或开始监控时，应明确：

- 当前目标是什么
- 下一步要做什么
- 后续会持续同步进展

### 2. 进展型输出
当阶段推进或达到节奏阈值时，应给出简短进展，例如：

- 当前 phase
- 最近完成了什么
- 接下来准备做什么

进展型输出应短、清晰、低噪音，不需要长篇推理。

### 3. 动作型输出
当 agent 做出重要决策时，应明确说明：

- 要采取什么动作
- 为什么这么做
- 是否需要用户介入

这样主 session 中会形成一条连续、可见、低噪音的行动轨迹。

## 后台执行与主 session 的边界

### 适合放后台的
- 长时间具体执行步骤
- 重计算 / 重 IO / 重外部交互
- 复杂分析

### 不适合放后台的
- 整体流程编排
- 关键策略判断
- 主要进展叙事
- 与用户讨论紧密耦合的持续决策

后台可以承担局部执行单元，但其结果应尽快回流主 session，由主 session 继续承担解释与编排职责。

## 推荐实现方式

### 1. 保留轻量触发层与状态层
保留以下组件：

- event/poll 获取增量
- state store 做去重与连续性支撑

但它们只负责输入与辅助，不负责主导闭环。

### 2. 改变决策流向
推荐流向应为：

`trigger -> 生成压缩输入 -> 主 session agent 决策 -> 在主 session 中输出进展 / 动作`

而不是：

`trigger -> 后台 watcher 自己判断 -> 最后再通知用户`

### 3. 主 session 作为默认进展出口
默认情况下，以下内容都应优先在主 session 中体现：

- 阶段推进
- 异常
- 提问
- 长时间 no-change 后的状态汇报
- 关键纠偏或升级动作

### 4. 重活仅作局部外包
如果某个动作确实很长：

- 可以临时外包给 subagent 或其他后台执行单元
- 但主 session 负责宣布开始、同步关键进展、解释最终结果

## 推荐流程

### 1. 用户提出目标
主 session 明确：

- 目标
- 预期阶段
- 汇报方式

### 2. 系统启动触发层
事件监听与定时巡查开始工作，但只负责采集增量，不负责主导完整闭环。

### 3. 每次触发整理为紧凑输入
输入内容可以包括：

- 最新 phase
- 最近 checkpoint
- 是否有 permission/question
- 是否连续 no-change
- 最近是否已做过可见更新

### 4. 主 session agent 进行判定
主 session 决定：

- 继续观察
- 输出进展
- 发起纠偏
- 追问用户
- 中断或停止
- 升级到后台执行/分析单元

### 5. 输出回到主 session
用户持续看到：

- agent 现在在做什么
- 当前是否有变化
- 是否需要自己介入
- agent 是否在根据讨论修正策略

### 6. 必要时局部外包
后台执行单元处理重任务，但关键结果和解释尽快回流主 session。

---

## 最终结论

最合理的方案不是把 watcher 放在系统正中央，也不是让后台黑盒承担完整执行过程，而是：

> **以主 session 作为执行与决策中心；让定时巡查和事件巡查只提供增量输入；让用户在主 session 中持续看到 agent 的进展、动作与修正；后台组件仅承担辅助输入与局部重执行。**
